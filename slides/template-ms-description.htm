Metadata

  • "data about data"
  • not actual text contents, rather like a library catalogue record or a page title of a book
  • 3 general types:
    • descriptive
    • structural
    • administrative
  • can be more or less formalized

What are they useful for?

  • supports the identification, retrieval, use, re-use, management, and preservation of information resources

  • increases the informational value of an object

  • when controlled by standards, supports interoperability and data exchange
  • can describe a collection, a single resource, or a component part of a larger resource

Two different user goals in mind

  • the librairian's goal: 
      - how to identify and describe "electronic publications"
  - => preferance for structured data
  • the scientific editor's goal: 
      - how to document encoding practices and cwork with digital resources
  - => combinaision of very fine grained tagging and loose descriptions

Strucural main parts

  • <fileDesc>: bibliographic description of an electronic file
  • <encodingDesc>: documents choices in the realm of editorial decisions
  • <profileDesc>: non-bibliographic aspects of the text (e.g. languages used)
  • <revisionDesc>: version history of the file itself
  • more recent: <xenoData>:
    • non TEI structured information (Dublin Core, EAD, MARCXML, MODS, OAI...)

a minimal header

<teiheader>
  <filedesc>
    <titlestmt>
      <title>A title?</title>
    </titlestmt>
    <publicationstmt>
      <p>Who published?</p>
    </publicationstmt>
    <sourcedesc>
      <p>Where from?</p>
    </sourcedesc>
  </filedesc>
</teiheader>

types of contents in the teiHeader

  • free prose
  • specialized groups of elements for structured info
  • declarations (e.g. <refsDecl>)
  • descriptions (e.g. <projectDesc>)
    • => two conflicting desires:
      • preserve the traditional free prose descriptions
      • structure fine grained data chuncks in a 'database manner'

<fileDesc>

  • mandatory elements:

    • <titleStmt>
    • <publicationStmt>
    • <sourceDesc>

  • optional elements:

    • <editionStmt>
    • <serieStmt>
    • <extent>
    • <notesStmt>

<titleStmt> example

<titlestmt>
  <title>Letter to my Children</title>
  <author xml:id="STC">Samuel Taylor Coleman</author>
  <editor xml:id="JL">John Lingus</editor>
  <principal>Mark Owen</principal>
  <respstmt>
	<resp>First encoding</resp>
	<name>Isabelle Basle</name> </respstmt>
</titlestmt>

<sourceDesc>

  • possible structures:
    • prose description: <p>
    • bibliographic description: <bibl>, <biblStruct>...
    • specialized elements: <msDesc>...
    • authority lists: <listPerson>, <listPlace>...

<encodingDesc>

  • encoding processes, choices, editiorial choices, conventions, categorization
  • prose or specialized elements:
    • <projectDesc>: goals or motives of the project
    • <editorialDecl>: editorial principles
    • <classDecl>: classification sytems
    • <tagDecl>: structured info about the usage of particular elements

<encodingDesc> example 1

<encodingdesc>
  <projectdesc>
   <p>Originally prepared for use in the production of a series of old-spelling
       concordances in 1968, this text was extensively checked and revised for use during the
       editing of the new Oxford Shakespeare (Wells and Taylor, 1989).</p>
  </projectdesc>
  <editorialdecl>
   <correction>
    <p>Turned letters are silently corrected.</p>
   </correction>
   <normalization>
    <p>Original spelling and typography is retained, except that long s and ligatured
         forms are not encoded.</p>
   </normalization>
  </editorialdecl>  
 </encodingdesc>

<profileDesc>

  • non bibliographic descriptions of the text 'profile'
  • <langUsage>: languages or writing systems used in the text
  • <textClass>: indexation in reference to a standard classification system
  • <particDesc>: information about the 'participants' (persons) cited or depicted in the text
  • <creation>: details of a text's creation, e.g. the date and place it was composed

<profileDesc> example

  </creation>

<revisionDesc>

  • very important: provides a journal of the main updates in the encoding
  • list of:
    • <change> @when, @who
  • can contain tags:
    • <persName>
    • <date> @when

<revisionDesc> example 

<revisiondesc>
 <listchange>
  <change when="1991-11-11" who="#LB"> deleted chapter 10 </change>
  <change when="1991-11-02" who="#MSM"> completed first draft </change>
 </listchange>
</revisiondesc>

<msDesc>

  • Manuscripts:
    • unique objects sometimes presenting complex features and with a great cultural value
    • require 'customized' descriptions (vs standardized conventions used modern printed books)
    • The recording of the physical aspect of the object and its history rely on various distinctions that need a flexible schema
    • used to record all kinds of text bearing objects (inscriptions, papyrii, etc.)
    • can also be used in the <text> element ('catalogue raisonnรฉ').

<msContents>

  • unstructured: several <p>...
  • structured: <summary>, <msItem>, <msItemStruct>

	<p>A collection of Lollard sermons</p>
</mscontents>

	</msitem>
	<msitem n="2">
		<locus>fol. 9r-16v</locus>

	</msitem>

<msItem>, <msItemStruct>

  • for each discrete item in a manuscript
  • can be categorise with @class
  • location in the manuscript indicated with <locus>
  • <author>, <respStmt>
  • <rubric>, <implicit>, <explicit>
  • <decoNote>, <filiation>, <textLang>
  • or nested <msItems>...

			
 <textlang mainlang="en" otherlangs="la">
 	Predominantly in English with Latin glosses
 </textlang>

<physDesc>

  • <objectDesc>: description of the (text bearing) object
  • <handDesc>: hands or styles (each in a <handBote>)
  • <decoDesc>: decorations (illustrations) (each in a <decoNote>)
  • <bidingDesc>: bindings
  • => descriptions can also be provided as prose or paragraphs
